The IBM PDF is ingested, chunked and embedded into ChromaDB. Each query now retrieves the most relevant chunks and injects them as context before calling the LLM.

What changed from Step 2
  • ChromaDB vector store introduced — the IBM PDF is chunked with a recursive splitter, embedded with OpenAI, and stored once at ingestion time.
  • Retrieval step inserted before the LLM call — every query is now embedded and matched against stored chunks by cosine similarity before the LLM sees it.
  • Context injection — top-k chunks are inserted into the prompt, giving the LLM access to document content it was never trained on.
  • The LLM is no longer the source of truth — it reasons over retrieved facts instead of recalling from memory, eliminating stale training-data answers.
  • UI gains a RAG toggle — Chat mode bypasses retrieval entirely, RAG mode routes through the new pipeline.

STEP 4 Deconstructing RAG — Hybrid Search

The vector-only retriever is replaced with a hybrid EnsembleRetriever combining BM25 keyword search and vector similarity at equal weights.

flowchart TB subgraph ING [" Ingestion — run once "] PDF[(IBM PDF)] -->|chunk + embed| VDB[(ChromaDB\nvector store)] end SB[Sidebar\ntemperature · max_tokens\nk · hybrid toggle\nchunk_size · overlap] -->|params| UI U([User]) --> UI[Streamlit UI] U -->|query| SPLIT{ } SPLIT -->|embed| VEC[Vector Retriever\ncosine similarity] SPLIT -->|keyword| BM25[BM25 Retriever\nterm frequency] VEC -->|top-k| ENS[EnsembleRetriever\n50% vector · 50% BM25] BM25 -->|top-k| ENS VDB -.->|index| VEC VDB -.->|all docs| BM25 ENS -->|merged chunks| CTX[Context Window] UI --> CTX CTX --> LLM[ChatOpenAI] LLM --> API[(OpenAI API\ngpt-4o)] API -->|stream| UI UI --> R([Response + Sources]) style BM25 fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style ENS fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style SB fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style PDF fill:#1a2e23,stroke:#374151,color:#6b7280 style VDB fill:#1a2e23,stroke:#374151,color:#6b7280 style VEC fill:#374151,stroke:#4b5563,color:#9ca3af style CTX fill:#374151,stroke:#4b5563,color:#9ca3af style U fill:#374151,stroke:#4b5563,color:#9ca3af style UI fill:#374151,stroke:#4b5563,color:#9ca3af style LLM fill:#374151,stroke:#4b5563,color:#9ca3af style API fill:#1a2e23,stroke:#374151,color:#6b7280 style R fill:#374151,stroke:#4b5563,color:#9ca3af style SPLIT fill:#0f1117,stroke:#0f1117,color:#0f1117
What changed from Step 3
  • Pure vector retrieval replaced — cosine similarity alone was returning high-level overview chunks instead of the specific data tables needed to answer exact questions.
  • BM25 retriever added in parallel — keyword search scores chunks by term frequency, guaranteeing exact matches (like "healthcare" or "retail") are never missed.
  • EnsembleRetriever merges both signals — BM25 and vector results are combined at 50/50 weight and deduplicated before being passed to the LLM.
  • Chunk size and overlap become configurable — sidebar sliders let you tune chunking strategy live without touching code.
  • Ingestion and the LLM call are unchanged — the improvement is entirely inside the retrieval layer.

STEP 5 Function Calling & Tools

The LLM is bound to two tools. It decides when to call them, with what inputs, and reasons over the results. Multi-step tool use is supported via a 5-round execution loop.

What changed from Step 4
  • Retrieval was always automatic — RAG mode injected chunks into every request whether they were relevant or not.
  • The LLM now decides when to retrieve — tools are bound via bind_tools() and the model chooses which tool to call, with what inputs, based on the query.
  • Multi-round execution loop — the model can call tools multiple times in sequence (up to 5 rounds) before producing a final answer, enabling compound queries like "search for cost → then calculate".
  • calculate_breach_cost tool added — arithmetic tool the LLM invokes once it has retrieved the necessary inputs from the knowledge base.
  • Tool call trace in UI — every tool name, input and output is shown in an expandable section, making the model's reasoning transparent.
  • The hybrid retrieval mechanism and ChromaDB index are unchanged — retrieval is now wrapped inside a tool rather than called automatically.

STEP 6 Improved Ingestion — Unstructured.io + GPT-4o Vision

The ingestion pipeline is replaced entirely. Unstructured classifies document elements by layout. Charts are rendered by pymupdf and described by GPT-4o vision. Garbled OCR is detected and rerouted automatically.

What changed from Step 5
  • PyPDFLoader removed — it produced a flat character stream, garbling or silently dropping charts, tables and multi-column layouts.
  • UnstructuredPDFLoader (hi_res) replaces it — uses computer vision layout analysis to classify each page region as Title, NarrativeText, Table, FigureCaption, etc.
  • Chart pages routed to GPT-4o vision — when a FigureCaption element or garbled OCR text is detected, pymupdf renders the full page as a PNG and GPT-4o describes the chart as an ImageSummary chunk.
  • Tables routed to GPT-4o text — raw table extractions are restructured into TableSummary chunks that preserve all numbers and relationships.
  • Garbled OCR detection — elements whose average token length is below 3.5 chars (bar chart label noise like "A oO A BB A") are automatically rerouted to vision instead of being stored as garbage text.
  • The retrieval layer, tool definitions, and execution loop are completely unchanged — the improvement is entirely in what was stored in ChromaDB at ingestion time.

STEP 8 Guardrails — Prompt Injection Detection

Every user message passes through a dedicated LLM classifier before reaching any pipeline. Messages scoring above 0.7 for injection risk are blocked and never reach the model, retriever, or tools.

What changed from Step 7
  • Hard guardrail added at the UI entry point — check_prompt_injection() runs before any mode handler. It is a separate LLM call that the main pipeline never influences.
  • Few-shot classification — the detector's system prompt contains labelled examples spanning 0.0 to 1.0. It generalises to novel attacks without a rules list.
  • Threshold at 0.7 — allows legitimate security questions through while blocking direct injection, persona hijacking, and defensive reframing attacks.
  • Soft guardrail replaced — the single system prompt line from Step 1 ("never provide instructions...") could be bypassed with context manipulation. The new guardrail is code, not a request.
  • All existing pipelines (Chat, RAG, Tools) are unchanged — the guardrail wraps them without touching their internals.

STEP 9 Observability & Fault Tolerance

Every LLM call is wrapped with structured logging, a 60-second timeout, and up to 3 retries with exponential backoff. Exhausted retries return a clean fallback message — never a Python traceback.

flowchart TB U([User]) --> UI[Streamlit UI] UI --> GUARD[Guardrail check] GUARD -->|pass| RWT[run_with_tools\n@log_llm_call] RWT --> RETRY[@with_retry\n3 attempts · exp backoff 1s→10s] RETRY --> TIMEOUT[invoke_with_timeout\n60s hard cap] TIMEOUT --> LLM[LLM Round N\nbind_tools] LLM -->|TimeoutError / APIError| RETRY RETRY -->|3 failures| FB([Fallback message]) LLM -->|success| UI UI --> R([Response]) style RWT fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style RETRY fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style TIMEOUT fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style FB fill:#7f1d1d,stroke:#ef4444,color:#fca5a5 style U fill:#374151,stroke:#4b5563,color:#9ca3af style UI fill:#374151,stroke:#4b5563,color:#9ca3af style GUARD fill:#374151,stroke:#4b5563,color:#9ca3af style LLM fill:#374151,stroke:#4b5563,color:#9ca3af style R fill:#374151,stroke:#4b5563,color:#9ca3af
What changed from Step 8
  • @log_llm_call decorator added — wraps any LLM function and logs the start time, elapsed duration, and any errors. Auto-detects async vs sync functions.
  • @with_retry() added via tenacity — 3 attempts with exponential backoff (1s, 2s, 4s, capped at 10s). Logs a WARNING before each retry sleep so activity is visible in the terminal.
  • invoke_with_timeout() added — asyncio.wait_for() imposes a 60-second hard cap on each LLM round. TimeoutError triggers the retry mechanism.
  • FALLBACK_MESSAGE returned on exhaustion — after 3 failures the user sees a clean message instead of a Python traceback. The fallback is applied at the outermost synchronous boundary.
  • The guardrail, tool definitions, MCP server, and retrieval pipeline are unchanged.

STEP 11 Simple Agentic Workflows — Sequential StateGraph

Three specialised agents run in a fixed sequence over shared state. Each reads what the previous wrote and adds its own output. The workflow — not the LLM — controls execution order.

What changed from Step 10
  • Single agent doing everything replaced — one agent decided its own investigation path, which could miss steps or conflate reasoning with research.
  • StateGraph with three specialised nodes — each agent has one role: extract facts (Node 1), research with tools (Node 2), synthesize a report (Node 3).
  • Shared state (TypedDict) passed between nodes — each node reads the full state and writes only its own field. LangGraph merges the return dict automatically.
  • Node 2 is itself a ReAct agent — workflow nodes can embed full agents. The workflow controls when it runs; within its turn it reasons autonomously.
  • Nodes 1 and 3 use pure LLM reasoning — no tools — keeping extraction and synthesis fast and deterministic.

STEP 12 Composing Multiple Agentic Workflows

Parallel analysis runs two agents simultaneously via asyncio.gather. A severity router then directs flow to either a Critical or Standard response agent. Code decides routing.

  • Fixed sequential order replaced — Step 11 ran every agent in the same path regardless of the incident.
  • Parallel execution via asyncio.gather — ThreatFeedbackAgent and ComplianceFeedbackAgent run concurrently in one node. Independent analyses that don't depend on each other run simultaneously, halving the time for this stage.
  • Conditional routing via add_conditional_edges — severity_router reads state["severity"] and returns "critical" or "standard". LangGraph routes to the corresponding node. Code decides routing.

An LLM supervisor decides at runtime which specialist sub-agents to invoke. Specialists are exposed as tools. The LLM — not code — decides routing based on incident content.

What changed from Step 12
  • Code-based routing replaced — Step 12 routed by severity string extracted from threat analysis. Fixed rules break down when routing depends on multiple overlapping factors.
  • LLM supervisor decides routing — the supervisor reasons about the incident content and calls only the specialists that are relevant, in the order it determines. No hardcoded conditions.
  • Specialists exposed as @tool functions — each specialist is a synchronous tool whose body spins up a full ReAct agent internally. From the supervisor's perspective they are identical to any other tool.
  • ThreadPoolExecutor for async nesting — specialist tools are sync functions called from inside an already-running async event loop. Each runs its async code in a fresh thread with its own event loop, avoiding RuntimeError.

STEP 14 Human-in-the-Loop

Low and medium severity incidents pause after Phase 1 and wait for human approval. Session state persists the pending proposal across Streamlit rerenders. Phase 2 runs only after a human decision.

What changed from Step 13
  • Two-phase separation introduced — Phase 1 gathers intelligence and assesses severity. Phase 2 compiles the final report. They are separate function calls that can run minutes apart.
  • Approval gate for Low / Medium / Unknown severity — the inverted threshold gates on ambiguity rather than stakes. A critical ransomware attack has a clear response; a low-severity accidental disclosure does not.
  • Session state as the pause mechanism — Python has no native way to pause mid-execution. The pending proposal is stored in st.session_state["hitl_pending"] and Phase 2 waits until a human decision arrives.
  • Forced rerun after pending state is set — st.rerun() fires after the Phase 1 message is appended, so the approval card renders on the next Streamlit cycle rather than being blocked by st.chat_input.
  • High and Critical severity incidents bypass the gate — they are auto-approved because speed matters and the response is generally clear-cut.

STEP 16 Agent-to-Agent (A2A)

A specialised ThreatIntelligenceAgent runs as a standalone FastAPI service on port 8888. The main app calls it over HTTP. Two completely independent processes — no shared code, no shared memory.

What changed from Step 15
  • All agents ran inside the Streamlit process — every previous step added capabilities within the same Python process. A2A breaks that constraint entirely.
  • Standalone FastAPI server introduced — threat_intel_server.py runs as a completely separate process on port 8888. It has its own LLM client, its own config, and can be restarted independently.
  • AgentCard published at /.well-known/agent-card.json — clients can discover what the agent does, what skills it has, and how to call it without reading source code.
  • HTTP/JSON protocol for agent communication — the main app calls POST /run with an incident string and receives an analysis string. No shared code path between client and server.
  • Online/offline detection — is_server_available() probes the server before every request. The sidebar shows green/red status and the chat handler returns a clear error if the server is down.
  • This is the final architectural evolution: local functions (Step 5) → in-process agents (Step 10) → distributed agents over HTTP (Step 16).
