STEP 1 Basic LLM Chat

The baseline. User messages stream through LangChain to GPT-4o and back. A system prompt defines the Aegis persona.

Baseline — starting point
  • Every user message goes directly to GPT-4o — no retrieval, no tools, no document access.
  • A SystemMessage is prepended on every request to define the Aegis persona and restrict responses to data breach analysis.
  • The model answers entirely from training data — stale figures, no cited sources.

STEP 3 Introduction to RAG

The IBM PDF is ingested, chunked and embedded into ChromaDB. Each query now retrieves the most relevant chunks and injects them as context before calling the LLM.

What changed from Step 2
  • ChromaDB vector store introduced — the IBM PDF is chunked with a recursive splitter, embedded with OpenAI, and stored once at ingestion time.
  • Retrieval step inserted before the LLM call — every query is now embedded and matched against stored chunks by cosine similarity before the LLM sees it.
  • Context injection — top-k chunks are inserted into the prompt, giving the LLM access to document content it was never trained on.
  • The LLM is no longer the source of truth — it reasons over retrieved facts instead of recalling from memory, eliminating stale training-data answers.
  • UI gains a RAG toggle — Chat mode bypasses retrieval entirely, RAG mode routes through the new pipeline.

STEP 4 Deconstructing RAG — Hybrid Search

The vector-only retriever is replaced with a hybrid EnsembleRetriever combining BM25 keyword search and vector similarity at equal weights.

flowchart TB subgraph ING [" Ingestion — run once "] PDF[(IBM PDF)] -->|chunk + embed| VDB[(ChromaDB\nvector store)] end SB[Sidebar\ntemperature · max_tokens\nk · hybrid toggle\nchunk_size · overlap] -->|params| UI U([User]) --> UI[Streamlit UI] U -->|query| SPLIT{ } SPLIT -->|embed| VEC[Vector Retriever\ncosine similarity] SPLIT -->|keyword| BM25[BM25 Retriever\nterm frequency] VEC -->|top-k| ENS[EnsembleRetriever\n50% vector · 50% BM25] BM25 -->|top-k| ENS VDB -.->|index| VEC VDB -.->|all docs| BM25 ENS -->|merged chunks| CTX[Context Window] UI --> CTX CTX --> LLM[ChatOpenAI] LLM --> API[(OpenAI API\ngpt-4o)] API -->|stream| UI UI --> R([Response + Sources]) style BM25 fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style ENS fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style SB fill:#1e40af,stroke:#3b82f6,color:#e0e0e0 style PDF fill:#1a2e23,stroke:#374151,color:#6b7280 style VDB fill:#1a2e23,stroke:#374151,color:#6b7280 style VEC fill:#374151,stroke:#4b5563,color:#9ca3af style CTX fill:#374151,stroke:#4b5563,color:#9ca3af style U fill:#374151,stroke:#4b5563,color:#9ca3af style UI fill:#374151,stroke:#4b5563,color:#9ca3af style LLM fill:#374151,stroke:#4b5563,color:#9ca3af style API fill:#1a2e23,stroke:#374151,color:#6b7280 style R fill:#374151,stroke:#4b5563,color:#9ca3af style SPLIT fill:#0f1117,stroke:#0f1117,color:#0f1117
What changed from Step 3
  • Pure vector retrieval replaced — cosine similarity alone was returning high-level overview chunks instead of the specific data tables needed to answer exact questions.
  • BM25 retriever added in parallel — keyword search scores chunks by term frequency, guaranteeing exact matches (like "healthcare" or "retail") are never missed.
  • EnsembleRetriever merges both signals — BM25 and vector results are combined at 50/50 weight and deduplicated before being passed to the LLM.
  • Chunk size and overlap become configurable — sidebar sliders let you tune chunking strategy live without touching code.
  • Ingestion and the LLM call are unchanged — the improvement is entirely inside the retrieval layer.

STEP 5 Function Calling & Tools

The LLM is bound to two tools. It decides when to call them, with what inputs, and reasons over the results. Multi-step tool use is supported via a 5-round execution loop.

What changed from Step 4
  • Retrieval was always automatic — RAG mode injected chunks into every request whether they were relevant or not.
  • The LLM now decides when to retrieve — tools are bound via bind_tools() and the model chooses which tool to call, with what inputs, based on the query.
  • Multi-round execution loop — the model can call tools multiple times in sequence (up to 5 rounds) before producing a final answer, enabling compound queries like "search for cost → then calculate".
  • calculate_breach_cost tool added — arithmetic tool the LLM invokes once it has retrieved the necessary inputs from the knowledge base.
  • Tool call trace in UI — every tool name, input and output is shown in an expandable section, making the model's reasoning transparent.
  • The hybrid retrieval mechanism and ChromaDB index are unchanged — retrieval is now wrapped inside a tool rather than called automatically.

STEP 6 Improved Ingestion — Unstructured.io + GPT-4o Vision

The ingestion pipeline is replaced entirely. Unstructured classifies document elements by layout. Charts are rendered by pymupdf and described by GPT-4o vision. Garbled OCR is detected and rerouted automatically.

What changed from Step 5
  • PyPDFLoader removed — it produced a flat character stream, garbling or silently dropping charts, tables and multi-column layouts.
  • UnstructuredPDFLoader (hi_res) replaces it — uses computer vision layout analysis to classify each page region as Title, NarrativeText, Table, FigureCaption, etc.
  • Chart pages routed to GPT-4o vision — when a FigureCaption element or garbled OCR text is detected, pymupdf renders the full page as a PNG and GPT-4o describes the chart as an ImageSummary chunk.
  • Tables routed to GPT-4o text — raw table extractions are restructured into TableSummary chunks that preserve all numbers and relationships.
  • Garbled OCR detection — elements whose average token length is below 3.5 chars (bar chart label noise like "A oO A BB A") are automatically rerouted to vision instead of being stored as garbage text.
  • The retrieval layer, tool definitions, and execution loop are completely unchanged — the improvement is entirely in what was stored in ChromaDB at ingestion time.
