What changed from Step 2
  • RAG Chain container added — sits between the UI and the LLM Client, coordinates retrieval and context injection before every query.
  • Embeddings Client added — a separate concern from the LLM Client: converts text to vectors for storage and query matching.

  • LLM Client no longer answers from memory — it receives a context window containing retrieved chunks before it generates any text.

STEP 4 Deconstructing RAG — Hybrid Search

The single vector retriever is replaced by a three-container retrieval stack: BM25, Vector, and an EnsembleRetriever that merges both signals.

What changed from Step 3
  • Single vector retriever replaced — one container doing cosine similarity alone missed exact keyword matches, surfacing overview chunks over specific data tables.
  • BM25 Retriever added — a new container that scores all stored documents by keyword term frequency. Never misses an exact match.
  • EnsembleRetriever added — combines and deduplicates results from both retrievers at configurable weights (default 50/50), giving the best of keyword and semantic search.
  • ChromaDB is now read twice per query: by the vector retriever (similarity index) and by BM25 (all documents loaded into memory for scoring).

What changed from Step 4
  • Tool Executor container added — run_with_tools() owns the multi-round loop: invokes the LLM, executes tool calls, feeds results back, repeats up to 5 times.
  • search_knowledge_base tool added — wraps the existing EnsembleRetriever as a callable tool. The LLM can now choose when to retrieve, rather than retrieval being automatic.
  • calculate_breach_cost tool added — pure arithmetic container. The LLM calls it once it has retrieved the required inputs from the knowledge base.
  • RAG was always-on — retrieval now only happens when the LLM decides a tool call is needed. Some queries are answered directly without touching ChromaDB.

What changed from Step 5
  • PyPDFLoader removed — produced a flat character stream, silently dropping charts and garbling table structures.
  • Unstructured Loader added — hi_res strategy uses computer vision layout analysis to classify every page region before extracting text.
  • Element Router added — dispatches each classified element to the right processor: NarrativeText goes to the splitter, Tables go to GPT-4o text, FigureCaptions and garbled OCR go to pymupdf + GPT-4o vision.
  • pymupdf Renderer added — renders full PDF pages (including vector graphics) to PNG images that GPT-4o vision can read.
  • GPT-4o Vision and GPT-4o Text containers added — produce ImageSummary and TableSummary chunks respectively, replacing garbled OCR output with accurate, retrieval-ready descriptions.
  • The Tool Executor, retrievers, and LLM Client are entirely unchanged — the improvement is purely in what was written to ChromaDB at ingestion time.
