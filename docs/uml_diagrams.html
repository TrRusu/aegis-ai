STEP 1 Basic LLM Chat

Baseline sequence: user sends a message, a SystemMessage is prepended, GPT-4o streams tokens back.

sequenceDiagram actor User participant UI as Streamlit UI participant LLM as LLM Client participant API as OpenAI API rect rgb(20, 50, 140) Note over User,API: Step 1 — full flow is new User->>UI: sends message UI->>LLM: build_messages(history + SystemMessage) LLM->>API: POST /chat/completions API-->>UI: token stream UI-->>User: streamed response end
Baseline — all interactions are new
  • build_messages() prepends a SystemMessage defining the Aegis persona on every call — the history list grows with each turn.
  • st.write_stream() on the UI side consumes the token generator and renders tokens as they arrive — the user sees output within ~1 second rather than waiting for the full response.
  • No retrieval, no tools. The model answers entirely from training data.

STEP 2 Model Parameters

A parameter negotiation step is added before the LLM call. Temperature and max_tokens now travel with every request.

sequenceDiagram actor User participant SB as Sidebar participant UI as Streamlit UI participant LLM as LLM Client participant API as OpenAI API rect rgb(20, 50, 140) Note over User,SB: New — user sets params before querying User->>SB: adjusts temperature, max_tokens SB-->>UI: params stored in session state end User->>UI: sends message rect rgb(20, 50, 140) Note over UI,LLM: New — LLM built dynamically with params UI->>LLM: build_llm(temperature, max_tokens) end LLM->>API: POST /chat/completions API-->>UI: token stream UI-->>User: streamed response
What changed from Step 1
  • Parameter negotiation step added — before any message is sent, the user interacts with the sidebar to set temperature and max_tokens.
  • build_llm() call added — the LLM client is now instantiated per request with the current parameter values rather than once at app startup.
  • All messages to OpenAI and the streaming response path are unchanged — only the configuration carried in the request payload changes.

STEP 3 Introduction to RAG

A retrieval loop is inserted before the LLM call. The query is embedded, matched against ChromaDB, and the top-k chunks are injected as context.

sequenceDiagram actor User participant SB as Sidebar participant UI as Streamlit UI participant RAG as RAG Chain participant EMB as Embeddings Client participant DB as ChromaDB participant LLM as LLM Client participant API as OpenAI API Note over DB: IBM PDF pre-ingested (chunks embedded + stored) User->>SB: adjusts params + selects RAG mode SB-->>UI: params + mode User->>UI: sends query rect rgb(20, 50, 140) Note over UI,DB: New — retrieval before LLM call UI->>RAG: build_rag_response(query, history, k) RAG->>EMB: embed(query) EMB->>API: POST /embeddings API-->>EMB: query vector EMB->>DB: similarity_search(vector, k) DB-->>RAG: top-k Document chunks RAG->>LLM: SystemMessage + context + query end LLM->>API: POST /chat/completions API-->>UI: token stream rect rgb(20, 50, 140) Note over UI,User: New — sources shown alongside response UI-->>User: response + source chunk expander end
What changed from Step 2
  • Retrieval loop inserted — query is embedded, cosine-searched against ChromaDB, and the top-k chunks are collected before the LLM is called.
  • Two OpenAI calls per turn — one to /embeddings for the query vector, one to /chat/completions for generation. Total latency increases but answers are grounded in real data.
  • Source expander in UI — the retrieved chunks are returned alongside the response so the user can verify what the model read.
  • Model no longer answers from memory — the LLM receives context injected into the system prompt; it cannot contradict what the retrieved text says without being explicitly told to.

STEP 4 Deconstructing RAG — Hybrid Search

The single similarity_search call is replaced by two parallel retrievals (BM25 + vector) that are merged by EnsembleRetriever.

sequenceDiagram actor User participant SB as Sidebar participant UI as Streamlit UI participant RAG as RAG Chain participant ENS as EnsembleRetriever participant VEC as Vector Retriever participant BM25 as BM25 Retriever participant DB as ChromaDB participant LLM as LLM Client participant API as OpenAI API User->>SB: sets params + hybrid mode + k SB-->>UI: params User->>UI: sends query UI->>RAG: build_rag_response(query, hybrid=true, k) rect rgb(20, 50, 140) Note over RAG,DB: New — two retrievers run in parallel, results merged RAG->>ENS: invoke(query) par Semantic retrieval ENS->>VEC: embed + cosine_search(query, k) VEC->>API: POST /embeddings API-->>VEC: query vector VEC->>DB: similarity_search(vector) DB-->>VEC: top-k semantic chunks VEC-->>ENS: semantic results and Keyword retrieval ENS->>BM25: keyword_search(query, k) BM25->>DB: load all documents DB-->>BM25: all chunks BM25-->>ENS: top-k keyword chunks end ENS-->>RAG: merged + deduplicated chunks end RAG->>LLM: SystemMessage + richer context + query LLM->>API: POST /chat/completions API-->>UI: token stream UI-->>User: response + sources
What changed from Step 3
  • Single similarity_search() removed — one cosine similarity call was surfacing general overview chunks that weren't specific enough for data questions.
  • Two parallel retrieval paths added — BM25 and vector run concurrently, each returning their top-k candidates.
  • EnsembleRetriever merge step added — combines both result sets with reciprocal rank fusion, deduplicates, and returns the best-ranked combined list to the RAG Chain.
  • BM25 requires loading all documents from ChromaDB into memory each time — this is a trade-off: slower per query but eliminates exact-match misses.

STEP 5 Function Calling & Tools

The fixed retrieval pipeline is replaced by a multi-round tool loop. The LLM decides when to search and when to calculate — shown here as a two-tool compound query.

sequenceDiagram actor User participant UI as Streamlit UI participant Chain as Tool Executor participant LLM as LLM Client (bind_tools) participant SKB as search_knowledge_base participant CBC as calculate_breach_cost participant ENS as EnsembleRetriever participant DB as ChromaDB participant API as OpenAI API User->>UI: "estimate breach cost for retail, 50k records" rect rgb(20, 50, 140) Note over UI,API: New — tool executor manages multi-round loop (up to 5) UI->>Chain: run_with_tools(prompt, history, k) Chain->>LLM: Round 1 — messages + tool definitions LLM->>API: POST /chat/completions API-->>LLM: tool_call: search_knowledge_base(query) LLM-->>Chain: [tool_calls] Chain->>SKB: invoke({query: "retail per-record cost"}) SKB->>ENS: hybrid retrieval ENS->>DB: BM25 + vector search DB-->>ENS: chunks ENS-->>SKB: top-k chunks SKB-->>Chain: ToolMessage(chunks) Chain->>LLM: Round 2 — + ToolMessage LLM->>API: POST /chat/completions API-->>LLM: tool_call: calculate_breach_cost(50000, 165) LLM-->>Chain: [tool_calls] Chain->>CBC: invoke({records_lost: 50000, cost_per_record: 165}) CBC-->>Chain: ToolMessage("USD 8,250,000") Chain->>LLM: Round 3 — + ToolMessage LLM->>API: POST /chat/completions API-->>LLM: final text (no tool_calls) LLM-->>Chain: response content Chain-->>UI: response + tool_calls_log UI-->>User: answer + tool trace expander end
What changed from Step 4
  • Fixed retrieval-then-generate pattern removed — in RAG mode, retrieval happened unconditionally on every request, even when not needed.
  • Multi-round tool loop added — the Tool Executor calls the LLM, checks for tool_calls, executes them, appends ToolMessages, and calls the LLM again. Repeats until the LLM returns a final text response or 5 rounds are exceeded.
  • Tool definitions sent upfront — the LLM receives JSON schemas for both tools in the first round so it can decide which (if any) to invoke based on the query.
  • calculate_breach_cost in the loop — the LLM can chain search → calculate in consecutive rounds, with each ToolMessage carrying the result forward as context.
  • The EnsembleRetriever and ChromaDB interactions are identical to Step 4 — they're now invoked inside search_knowledge_base rather than the RAG Chain.

What changed from Step 5
  • PyPDFLoader replaced — the one-time ingestion now runs a full pipeline: layout analysis → element classification → per-type processing.
  • Three ingestion paths added — Tables go to GPT-4o text, chart pages go to pymupdf → GPT-4o vision, NarrativeText goes to the splitter. Each produces a different chunk category in ChromaDB.
  • Garbled OCR detection — the Element Router checks average token length; elements below 3.5 chars/token (OCR noise from chart axes) are automatically rerouted to vision.
  • The runtime sequence (right side) is byte-for-byte the same as Step 5 — except search_knowledge_base now retrieves ImageSummary and TableSummary chunks that actually contain the chart data that was previously missing.
